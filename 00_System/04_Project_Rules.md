# Project Rules

## Purpose
Encodes the mandatory, non-negotiable rules that govern every requirement, artifact, and analysis produced under this repository — carried over from the operating standards used across Target Group / Uranium Technology Solutions engagements.

## Scope
Applies to all discovery, requirements engineering, BRD/FRD creation, backlog management, and process design work.

## Intended Usage
### Rule 1 — No Assumptions
Never invent, assume, estimate, infer, hallucinate, or fabricate business requirements, rules, workflows, approval cycles, roles, permissions, calculations, validations, notifications, reports, integrations, screens, fields, or data elements. Use `MISSING INFORMATION`, `CLARIFICATION REQUIRED`, or `ASSUMPTION NOT ALLOWED` when information is missing.

### Rule 2 — Source-Driven Analysis
Use only information explicitly available in meeting minutes, BRDs, process documents, existing systems, screenshots, user manuals, policies, or uploaded files. Every requirement must be traceable to a source.

### Rule 3 — Branding Neutrality
Project branding (Target vs. Uranium Technology Solutions) must always be determined from source documents, project context, or explicit instruction — never assumed. Default to branding-neutral until clarified.

### Rule 4 — Traceability
Preferred source hierarchy: (1) Approved Business Documents, (2) Discovery Sessions, (3) Meeting Minutes, (4) Client Communications, (5) Existing Systems, (6) User Clarifications. Flag items where traceability cannot be established.

### Rule 5 — Requirement Quality
Every requirement is evaluated for Clarity, Completeness, Consistency, Traceability, and Testability before acceptance.

## Future Expansion
Rules will be versioned as engagement scope expands (e.g. regulatory-specific rule sets for pharma, banking, telecom clients).
